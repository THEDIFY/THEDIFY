## üöÄ How to Contribute

### 1. **Code Contributions**

#### Getting Started
```bash
# Fork the repository
# Clone your fork
git clone https://github.com/YOUR_USERNAME/THEDIFY.git
cd THEDIFY

# Create a feature branch
git checkout -b feature/your-feature-name

# Make your changes
# ...

# Commit with clear messages
git commit -m "feat: add [feature description]"

# Push to your fork
git push origin feature/your-feature-name

# Open a Pull Request
```

#### Code Standards
- **Python:** Follow PEP 8, use Black formatter
- **JavaScript/TypeScript:** Follow Airbnb style guide, use ESLint + Prettier
- **Commits:** Use conventional commits (`feat:`, `fix:`, `docs:`, `test:`, etc.)
- **Documentation:** Update README and relevant docs with your changes
- **Tests:** Add tests for new features (aim for 80%+ coverage)

### 2. **Documentation Contributions**

We welcome improvements to:
- Project READMEs
- Code comments and docstrings
- Reproducibility guides
- Architecture diagrams
- Tutorial content

**How to Contribute Docs:**
- Fix typos, clarify instructions, add examples
- Create visual aids (diagrams, screenshots)
- Write tutorials for common use cases
- Translate documentation (multi-language support)

### 3. **Bug Reports**

**Before Reporting:**
- Search existing issues to avoid duplicates
- Test on the latest version
- Gather reproduction steps

**What to Include:**
```markdown
### Bug Description
Clear description of the issue

### Steps to Reproduce
1. Step one
2. Step two
3. ...

### Expected Behavior
What should happen

### Actual Behavior
What actually happens

### Environment
- OS: [e.g., Ubuntu 22.04]
- Python: [e.g., 3.11.5]
- Project: [EDIFY / Axolotl / GUIRA / MenTora]
- Version: [e.g., v1.2.0]

### Screenshots/Logs
If applicable
```

### 4. **Feature Requests**

**Before Requesting:**
- Check if it aligns with project mission
- Search existing feature requests
- Consider implementation complexity

**What to Include:**
- **Problem:** What problem does this solve?
- **Solution:** Proposed approach
- **Alternatives:** Other options considered
- **Impact:** Who benefits and how?

## üìã Pull Request Process

1. **Before Submitting:**
   - [ ] Code follows style guidelines
   - [ ] All tests pass (`pytest` / `npm test`)
   - [ ] Documentation updated
   - [ ] Commits are clean and descriptive
   - [ ] Branch is up-to-date with `main`

2. **PR Template:**
   ```markdown
   ## Description
   Brief description of changes

   ## Motivation
   Why is this change needed?

   ## Changes
   - Change 1
   - Change 2

   ## Testing
   How was this tested?

   ## Screenshots
   If UI changes

   ## Checklist
   - [ ] Tests added/updated
   - [ ] Documentation updated
   - [ ] Code follows style guide
   ```

3. **Review Process:**
   - Maintainer will review within 3-5 business days
   - Address feedback in new commits
   - Once approved, maintainer will merge
